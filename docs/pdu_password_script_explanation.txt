Raritan PDU Password Bootstrap Script - What It Does
===================================================

File being documented:

    bootstrap_pdu_passwords.py

Purpose
-------

This script automates the first password setup step for Raritan PDUs.

It reads a list of PDU management IP addresses, logs in to each PDU as the
admin user, and changes the admin password from the current/default password
to a new password.

This is intended to replace the manual first-login password-change step when
bringing up multiple PDUs.

What the script changes
-----------------------

The script changes only this account:

    Username: admin
    Password: whatever new password you enter when the script runs

What the script does NOT change
-------------------------------

This script does not change outlet power.
It does not turn outlets on or off.
It does not reboot the PDU.
It does not update firmware.
It does not change network settings.
It does not configure SNMP, syslog, NTP, DNS, users, roles, or names.
It does not create new accounts.
It does not delete accounts.
It does not store the new password in a file.

High-level flow
---------------

For each PDU IP address, the script does this:

1. Try logging in as admin using the new password.

   This makes the script safe to rerun.

   If the new password already works, the script reports:

       OK - already_changed

   Then it moves on to the next PDU.

2. If the new password does not work, try the old/current/bootstrap password.

   For brand-new PX3-style Raritan PDUs, the default password is commonly:

       raritan

   The script uses admin as the username.

3. If the PDU requires a first-login password change, the Raritan API may
   return HTTP 451.

   The script treats HTTP 451 as expected. It means the PDU is reachable and
   the admin account needs its password changed before normal login can
   continue.

4. The script calls the Raritan API method that changes the admin password.

   The method used is:

       setAccountPassword()

   The user resource used is:

       /auth/user/admin

5. If the password change succeeds, the script verifies the result.

   It logs in again using the new password.

   If that works, the script reports:

       OK - changed

6. If something fails, the script reports an error for that IP address and
   continues to the next PDU.

Input file
----------

The default IP list file is:

    pdus.txt

The script expects one PDU IP address per line.

Example pdus.txt:

    10.42.98.10
    10.42.98.11
    10.42.98.12
    10.42.98.13
    10.42.98.14

Blank lines are ignored.
Lines that start with # are ignored.
Duplicate IP addresses are skipped.

Example with comments:

    # Raritan PDUs for cabinet BJ16-BJ25
    10.42.98.10
    10.42.98.11

    # Spare / test unit
    10.42.98.12

Passwords
---------

The script needs two passwords:

1. The old/current/bootstrap admin password
2. The new admin password you want to set

The old password can be provided three ways:

1. By typing it when prompted
2. By setting the environment variable PDU_OLD_PASSWORD
3. By using --old-password on the command line

The safest normal method is to type it when prompted.

Passing a password on the command line is supported, but it is less safe
because command-line arguments can be visible in shell history and process
listings.

The new password is typed interactively when the script runs.
The script asks for it twice to avoid typos.

The script does not print the passwords while you type them.

Password checks
---------------

Before changing anything, the script checks that:

    The new password and confirmation match.
    The new password is at least the configured minimum length.
    The new password is not the same as the old/current password.

The default minimum length is:

    8 characters

You can change that local check with:

    --min-length NUMBER

The PDU may have stronger password rules than the script's local check.
If the PDU rejects the password, the script tries to translate the API return
code into a readable result.

Examples:

    password_too_short
    password_too_long
    password_empty
    password_needs_lowercase
    password_needs_uppercase
    password_needs_number
    password_needs_special
    password_in_history
    password_too_short_for_snmp
    password_has_control_chars

Dry-run mode
------------

Dry-run mode checks whether the script can reach each PDU using the old/current
password. It does not change any passwords.

Run dry-run mode with:

    --dry-run

In dry-run mode, a successful result looks like:

    OK - would_change

That means the script reached the PDU with the old/current password and would
attempt to change the password during a live run.

Dry-run mode is the recommended first test.

Concurrency
-----------

The script can process multiple PDUs in parallel.

The option is:

    --concurrency NUMBER

The default is:

    1

For first use, leave it at 1.

After testing, using a higher value such as 4 is reasonable for a small batch.

The script blocks invalid values like 0 or negative numbers.

TLS certificate behavior
------------------------

By default, the script does not verify the PDU HTTPS certificate.

That is intentional.

Default behavior:

    TLS certificate verification is disabled.

If you have valid trusted certificates on the PDUs and want strict checking,
run with:

    --no-insecure

Logging
-------

By default, the script prints progress to the console.

You can also write a log file with:

    --log-file pdu-bootstrap.log

You can enable debug logging with:

    -v

or:

    --verbose

Example:

    python .\bootstrap_pdu_passwords.py --ips .\pdus.txt --dry-run -v

Exit codes
----------

The script exits with these codes:

    0     All PDUs processed successfully
    1     Local setup/input problem, such as no IPs, bad password confirmation,
          or invalid concurrency
    2     One or more PDUs failed
    130   The script was interrupted with Ctrl+C

Common results
--------------

OK - would_change

    Dry-run succeeded. The script would attempt to change this PDU during a
    live run.

OK - changed

    The script changed the admin password and verified that the new password
    works.

OK - already_changed

    The PDU already accepted the new password. Nothing needed to be changed.
    This is useful when rerunning the script.

OK - password_unchanged

    The PDU reported that the password was unchanged. This is treated as OK,
    but you should review whether that was expected.

ERROR - HTTP Error 401 or 403

    The password used for that PDU is probably wrong, or the admin account is
    not accepting that password.

ERROR - connection timeout / connection refused / DNS failure

    The script could not reach the PDU at that IP address.

ERROR - password_needs_special or similar

    The new password does not satisfy the PDU password policy.

Important assumptions
---------------------

The script assumes:

    The PDU is reachable over HTTPS.
    The username is admin.
    The PDU supports the Raritan/Xerus JSON-RPC API.
    The Python raritan package is installed.
    The person running the script knows the old/current/bootstrap admin password.
    The IP list file exists and contains the correct PDU management IPs.

Safety notes
------------

Run dry-run first.

Use --concurrency 1 the first time.

Keep the IP list small and controlled.

Do not put production passwords in shared notes, tickets, screenshots, or chat.

Prefer the interactive password prompt instead of passing passwords on the
command line.

Save a log file for audit records, but remember that the log file should not
include passwords.

Suggested first test
--------------------

Use one PDU in pdus.txt first.

Then run:

    python .\bootstrap_pdu_passwords.py --ips .\pdus.txt --dry-run -v

If that looks good, run live:

    python .\bootstrap_pdu_passwords.py --ips .\pdus.txt --log-file .\pdu-bootstrap.log

After that works on one PDU, add the remaining PDUs to pdus.txt.
